# CapiRogue 개발 계획

업데이트: 2026-04-29

## 1. 현재 목표

CapiRogue를 로컬에서 실행 가능한 Vite + React 게임 프로토타입으로 안정화한다. 1차로 주요 UI 문구와 게임 흐름은 정리했으므로, 다음 목표는 Node.js 설치 후 실제 실행 검증, 플레이 루프 점검, 저장/밸런스/UI 보강이다.

## 2. 현재 상태

- 프로젝트는 Vite + React 구조다.
- 전역 상태는 `zustand` 기반의 `src/store/useGameStore.js`가 담당한다.
- 화면 전환은 `src/App.jsx`에서 `screen` 값에 따라 처리한다.
- 주요 화면은 로그인, 타이틀, 조언자 선택, 메인, 이벤트, 정산, 결과, 보상, 게임오버로 구성되어 있다.
- 2026-04-29 기준으로 주요 한글 UI 문구와 게임 흐름 안내를 1차 정리했다.
- 전략 미리보기는 운영 선택을 먼저 반영한 뒤 시장 예측을 계산하도록 수정했다.
- 현재 터미널에서 `node`, `npm` 명령어가 잡히지 않아 실행/빌드 검증은 아직 못 했다.

## 3. 1차 반영 완료

- 로그인 화면을 로컬 프로필/게스트 시작 흐름으로 정리했다.
- 로컬 프로필 로그인 시 localStorage 저장 데이터를 불러오도록 연결했다.
- 상태바를 `월차`, `경기`, `체력`, `크레딧`, `흐름`, `보상`, `자문가` 기준으로 정리했다.
- 크레딧 버튼을 `회복`, `수요`, `견제`로 바꾸고 크레딧이 없으면 비활성화했다.
- 오른쪽 전략 패널에 이번 달 예상 생산량, 판매량, 매출, 비용, 자본 변화를 표시했다.
- 외부 이벤트는 `시장 뉴스`, 내부 이벤트는 `돌발 상황`으로 구분했다.
- 정산 화면에 최종 손익, 시장점유율, 생산량, 판매량, 비용 구조, 자본 변화를 정리했다.
- 결과 화면에 다음 보상까지 남은 기간과 다음 행동 버튼을 표시했다.
- 보상 화면에 보상 타이밍과 선택 이후 흐름을 설명했다.
- 게임오버 화면에 최고 도달 월차, 최고 월손익, 최근 기록, 레거시 수를 추가했다.
- 중후반 라이벌 슬롯명이 중복되지 않도록 활성 라이벌 순서대로 번호를 붙였다.
- 사용자 화면에서는 `floor`를 `월차/개월` 중심으로 표현하도록 바꿨다.

## 4. 남은 핵심 작업

### 4-1. 실행 환경 준비

Node.js LTS 설치 후 VS Code를 다시 열고 아래 명령을 확인한다.

```powershell
node --version
npm --version
```

의존성 설치와 개발 서버 실행:

```powershell
cd "C:\Users\ASUS\OneDrive - 제주대학교\바탕 화면\개발\캐피로그3"
npm install
npm run dev
```

완료 기준:

- Vite 개발 서버 주소가 표시된다.
- 브라우저에서 앱이 열린다.
- 콘솔에 치명적인 JSX/runtime 오류가 없다.

### 4-2. 기본 플레이 루프 검증

확인할 흐름:

1. 로컬 프로필 또는 게스트 시작
2. 타이틀 화면 진입
3. 조언자 선택
4. 게임 시작
5. 시장 뉴스 발생 여부 확인
6. 가격, 품질, 운영 전략 선택
7. 전략 확정
8. 돌발 상황 발생 여부 확인
9. 월말 정산 확인
10. 결과 화면 확인
11. 보상 또는 다음 달 진행
12. 게임오버 또는 120개월 클리어 확인

완료 기준:

- 최소 10개월 이상 연속 진행이 가능하다.
- 화면 전환 중 빈 화면이나 멈춤이 없다.
- 정산 전 예상값과 실제 정산값이 크게 어긋나지 않는다.

### 4-3. 저장/이어하기 흐름 결정

현재 반영:

- 로컬 프로필 로그인 시 localStorage 저장 데이터를 불러온다.
- 계정 모드에서 런 종료 시 레거시 정보가 저장된다.
- 게스트 모드는 저장 없는 플레이로 유지된다.

다음 결정:

- 타이틀 화면에 `이어하기`가 필요한지 정한다.
- localStorage 저장 범위를 레거시만 둘지, 진행 중 런까지 저장할지 정한다.
- `확인 코드` 입력이 실제로 필요한지, 프로필 이름만으로 충분한지 결정한다.

### 4-4. 밸런스 점검

확인할 항목:

- 초기 자본 6000, 부채 1800이 적절한지
- 공장 업그레이드 비용과 효과가 과하거나 약하지 않은지
- 가격 전략별 생산량 비율이 이해 가능한지
- 5개월 보상 주기가 너무 빠르거나 느리지 않은지
- 체력 감소 속도와 10개월마다 회복이 적절한지
- 크레딧 사용 효과가 체감되는지
- 120개월 클리어가 현실적인 난이도인지

완료 기준:

- 초반 10개월이 지나치게 쉽거나 어렵지 않다.
- 가격/품질/운영 선택마다 장단점이 보인다.
- 손실 원인이 정산 화면에서 납득된다.

### 4-5. UI/UX 확인

확인할 항목:

- 상태바가 1366px 화면에서 너무 빽빽하지 않은지
- 오른쪽 전략 패널의 예측 박스가 잘 읽히는지
- 모바일 또는 좁은 화면에서 버튼과 텍스트가 겹치지 않는지
- 보상 카드와 이벤트 카드의 문구가 잘리지 않는지
- 크레딧 버튼의 비활성 상태가 명확한지

완료 기준:

- 데스크톱 기준으로 핵심 조작이 한눈에 보인다.
- 모바일에서도 진행 버튼과 선택지가 화면 밖으로 밀리지 않는다.

### 4-6. 빌드 검증

실행 명령:

```powershell
npm run build
```

완료 기준:

- 빌드가 성공한다.
- 사용하지 않는 import, JSX 오류, 런타임 오류가 없다.
